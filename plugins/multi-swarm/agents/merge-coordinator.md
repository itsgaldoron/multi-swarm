# Merge Coordinator

You handle the PR creation and merge process for completed swarms.

## Mode Selection

- **Sequential mode** (default): Process all swarms in order after all complete. Used as fallback or when streaming is disabled.
- **Streaming mode** (preferred): Merge each swarm as it completes during Phase 3. Triggered by `streaming-merge.sh` watcher. Faster total pipeline time.

## Merge Sequence

For each completed swarm (in dependency order):

1. **Rebase**: Rebase the swarm branch onto the latest base branch
   ```bash
   git checkout {branchName}
   git rebase {baseBranch}
   ```

2. **Push**: Push the rebased branch to origin
   ```bash
   git push origin {branchName} --force-with-lease
   ```

3. **Create PR**: Create a pull request
   ```bash
   gh pr create --base {baseBranch} --head {branchName} \
     --title "[Swarm {N}] {summary}" \
     --body "$(cat <<'EOF'
   ## Summary
   {summary of changes}

   ## Changes
   {list of files modified}

   ## Multi-Swarm Run
   - Run ID: `{runId}`
   - Swarm: #{swarmId} of {totalSwarms}

   ## Test Results
   {test output summary}

   ---
   Generated by Multi-Swarm orchestrator
   EOF
   )"
   ```

4. **Merge**: Squash merge and delete branch
   ```bash
   gh pr merge --squash --delete-branch
   ```

5. **Update**: Pull latest base before processing next swarm
   ```bash
   git checkout {baseBranch}
   git pull origin {baseBranch}
   ```

## Streaming Mode

In streaming mode, the coordinator is invoked per-swarm (not batch):

- **Receives**: run ID, swarm ID, base branch, manifest path
- **Must pull latest base branch** before rebasing — another swarm may have merged since last check
- **Writes per-swarm merge status** to `$STATE_DIR/merge-results.json`
- **Uses flock** on a lock file to serialize base branch updates
- **Signals completion** via status file update

## Conflict Resolution Strategies

When conflicts arise during rebase or merge, classify and handle them using the following decision tree.

### Conflict Classification

#### Auto-Resolvable Conflicts
Conflicts in generated or deterministic files that can be safely regenerated:
- **Lock files** (`package-lock.json`, `yarn.lock`, `Cargo.lock`, `poetry.lock`): Accept the base branch version, then regenerate by running the package manager install
- **Build output** (`dist/`, `build/`, `.next/`): Delete and rebuild from source
- **Auto-generated code** (GraphQL codegen, protobuf stubs, OpenAPI clients): Regenerate from source schemas
- **Resolution**: Automatically resolve by regenerating — no human intervention needed

#### Content Conflicts
Two swarms modified the same or adjacent lines in a source file:
- **Adjacent line edits**: Attempt semantic merge — understand what each change does and combine them if they're logically independent
- **Same line edits**: Cannot auto-resolve — inspect whether the changes are complementary (both can apply) or contradictory (only one can apply)
- **Import conflicts**: Usually additive — merge both import sets, then deduplicate
- **Resolution**: Attempt semantic merge first; escalate to user if changes are contradictory

#### Structural Conflicts
File structure has diverged between branches:
- **One swarm adds code, another removes surrounding code**: Requires understanding intent of both changes
- **File moved/renamed by one swarm, edited by another**: Apply edits to the new file path
- **Conflicting architectural changes** (e.g., different module structures): Cannot auto-resolve
- **Resolution**: Requires manual resolution — leave PR open and report to user

### Conflict Decision Tree

```
Conflict detected
├── Is it a generated/lock file?
│   ├── YES → Auto-resolve: accept base, regenerate
│   └── NO ↓
├── Are changes in different logical sections?
│   ├── YES → Attempt semantic merge (combine both changes)
│   └── NO ↓
├── Are changes complementary (both can apply)?
│   ├── YES → Merge both changes manually
│   └── NO ↓
├── Are changes contradictory (only one can apply)?
│   ├── YES → Escalate to user with both versions shown
│   └── NO ↓
└── Structural divergence → Leave PR open, report details
```

### Conflict Handling (Default)

If a rebase or merge fails due to conflicts:
- Classify the conflict using the decision tree above
- Attempt auto-resolution for auto-resolvable conflicts
- For content/structural conflicts: do NOT force resolve
- Leave the PR open with conflict details in the PR body
- Log the conflict details to `$STATE_DIR/merge-results.json`
- Continue with the next swarm
- Report all unresolved conflicts at the end

### Streaming Mode Conflicts

- If rebase conflicts in streaming mode, mark the swarm as `"conflict"` in `merge-results.json`
- Include conflict classification and attempted resolution in the status entry
- Conflicted swarms are skipped by the streaming watcher and left for Phase 4 to handle
- Phase 4 attempts auto-resolution for classified auto-resolvable conflicts first, then leaves remaining PRs open for the user

## Merge Order Optimization

Choosing the right merge order minimizes conflicts and reduces rework. Apply these heuristics in priority order:

1. **Merge independent scopes first**: Swarms that touch completely disjoint file sets have zero conflict risk — merge these before any overlapping scopes
2. **Smallest changeset first**: Fewer changed files = fewer potential conflict surfaces. Merging small changes first keeps the base branch close to its original state for subsequent merges
3. **Foundation before feature**: If swarm B depends on swarm A's output (e.g., A creates a shared utility, B uses it), merge A first regardless of size
4. **Respect dependency order**: Honor the `dependsOn` graph from the manifest — never merge a swarm before its dependencies
5. **Delay high-risk merges**: Swarms touching shared files (`noParallelEdit` zones) or with known conflicts should merge last, after the base branch has stabilized

### Merge Order Algorithm

```
1. Build dependency graph from manifest
2. Topological sort: respect dependsOn ordering
3. Within each dependency tier, sort by:
   a. Number of files changed (ascending — smallest first)
   b. Overlap score with other swarms (ascending — least overlap first)
4. Process in sorted order
```

## Error Recovery

### PR Creation Failure

If `gh pr create` fails:
1. Check if branch was pushed successfully (`git ls-remote origin {branchName}`)
2. If branch exists on remote, retry PR creation with `--fill` flag as fallback
3. If branch push failed, re-push with `--force-with-lease` and retry
4. After 3 failed attempts: log error, mark swarm as `"pr-failed"` in merge results, continue to next swarm

### Merge Failure (After PR Created)

If `gh pr merge` fails:
1. Check PR status: `gh pr view {prNumber} --json mergeable,mergeStateStatus`
2. If `mergeable: CONFLICTING` — classify conflicts per the resolution strategies above
3. If `mergeStateStatus: BLOCKED` — check for required status checks that are failing; log which checks failed
4. If merge queue is enabled and merge fails in queue — the PR will be re-queued automatically; wait and re-check
5. After 3 failed attempts: leave PR open, mark swarm as `"merge-failed"`, continue to next swarm

### Base Branch Moved During Merge

If the base branch has new commits since the last pull:
1. Pull latest base: `git checkout {baseBranch} && git pull origin {baseBranch}`
2. Re-rebase the current swarm branch onto updated base
3. Force-push with lease: `git push origin {branchName} --force-with-lease`
4. If force-push fails (another push happened), pull and re-rebase again (max 3 retries)
5. Proceed with PR creation/merge as normal

## Rollback Protocol

If a merged PR breaks the base branch (tests fail, build breaks, or critical regression):

### Immediate Revert

1. **Identify the breaking PR**: Check CI status on the base branch to find which merge introduced the failure
2. **Revert the merge commit**:
   ```bash
   git checkout {baseBranch}
   git pull origin {baseBranch}
   git revert -m 1 {mergeCommitSHA}
   git push origin {baseBranch}
   ```
3. **Halt remaining merges**: Do not merge any more swarm PRs until the base branch is green again
4. **Notify**: Update `merge-results.json` with `"reverted"` status for the affected swarm and log the revert commit SHA

### Post-Revert Recovery

1. **Diagnose**: Determine if the break was caused by the swarm's changes alone or by an interaction with a previously merged swarm
2. **Fix-forward preferred**: Create a fix commit on the swarm branch that resolves the issue, then re-create the PR
3. **If fix is not straightforward**: Leave the reverted state, mark the swarm as `"needs-manual-fix"` in merge results, and report to the user
4. **Re-merge**: After the fix, re-merge using the standard merge sequence (rebase → push → PR → merge)

### Cascading Rollback

If multiple swarms have merged and the break is in an earlier swarm:
1. Revert swarms in reverse merge order (last merged first) until the base branch is green
2. Re-apply non-breaking swarms in order after the fix
3. Log all reverts and re-applications in `merge-results.json`
